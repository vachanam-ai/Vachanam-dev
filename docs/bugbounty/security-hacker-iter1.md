# Security Hacker — Iteration 1 Findings (Vachanam end-to-end audit)

**Scoreboard:** 41 findings — 4 CRITICAL · 11 HIGH · 16 MEDIUM · 10 LOW — claimed tier: **In the race (≥15)**, reaching toward Top 10.

Authorized whitehat audit of the owner's own product. READ-ONLY — no code changed. Every finding has a `file:line` and a concrete attack/impact. FIXLOG #1–105 cross-checked; already-fixed items are NOT re-reported except where a deploy/config artifact regresses a fix (e.g. #15 below regresses FIXLOG #79).

Two voice stacks coexist in the tree: the **Pipecat/Vobiz** path (`agent/server.py` + `agent/bot.py`) and the **LiveKit** path (`agent/livekit_minimal/agent.py`, the current production brain per memory `project-livekit-v2-working`). The LiveKit path has the bounce-round multi-tenant guards; the Pipecat path does **not** and is still wired (`agent/server.py` is the agent Dockerfile CMD). Several CRITICAL/HIGH findings live in the still-shipped Pipecat path.

## Do NOT weaken RULE 1/2/3/4/5 while fixing

- **RULE 1 (tenant isolation):** every fix must keep `branch_id` scoping on every read/write/cache key/calendar/log. Do not "simplify" the >1-branch refusal in `agent/livekit_minimal/agent.py:1064` — extend the SAME guard to the Pipecat path (#2), never remove it.
- **RULE 2 (atomic token):** when fixing the slot double-book residue (#18), add a DB/Redis serialization — never fall back to a DB-count for the next number.
- **RULE 3 (held token dies with the call):** keep the disconnect/shutdown DECR-rollback (`_cleanup_on_shutdown`, `release_token_on_disconnect`) intact when reworking the booking tools.
- **RULE 4 (calendar write part of booking, notifications never block it):** do not make the SSRF/webhook fix (#7) cause calendar writes to silently skip; keep notification failures non-blocking.
- **RULE 5 (branch from the dialed DID, never the caller):** fixes #2/#4/#13 must resolve the branch from a *trusted* carrier-side DID, not the caller-settable SIP/start-event field. Locking the trunk and validating the DID is the fix — do not paper over it by trusting the caller.

Full detail above; nothing in the codebase was modified by this audit.
