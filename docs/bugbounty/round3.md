## Verified non-issues (round-2 fixes inspected — genuinely correct)

- **Calendar event timezone (booking create/update)** — `create_booking_event`/`update_event` send a NAIVE local datetime plus `"timeZone": "Asia/Kolkata"` (calendar_service.py:141-148, 314-323). Google interprets the wall-clock in that tz, so a 15:00 IST booking lands at 15:00 IST on the calendar (NOT 15:00 UTC). booking_tools builds the datetime from the branch-local date+time. Correct — no tz shift.
- **Calendar event PII (Rule 9)** — summary is `"Apt — {first_name} (xx{last4})"`, description is `""` (calendar_service.py:139-140). No complaint/health/full-phone. Compliant.
- **C1 different_person + sequential double-confirm** — for `different_person=true` the re-check at booking_tools.py:786 correctly rejects the 2nd patient into a max=1 slot; a single call confirming twice is caught by the phone dup guard (734-746). Only the *concurrent* assign-skip path leaks (T1).
- **confirm capacity re-check off-by-one** — re-check counts only `status='confirmed'` DB rows (the new token isn't inserted/committed yet) and applies `exclude_token_id` on reschedule, so `slot_confirmed >= max` and `confirmed_count >= daily_token_limit` are correct bounds. No off-by-one.
- **_release_hold token-vs-slot guard** — only DECRs keys starting `slot:` (agent.py:657), never token counters (the counter IS the queue sequence). Matches the cancel/disconnect rule. Correct for the in-band failure paths it covers (the gap is the disconnect window — T4).
- **trial_pause job** — selects `status='trial' AND trial_ends_at IS NOT NULL AND trial_ends_at < now()` (UTC, comparing tz-aware columns), flips to paused, commits, then best-effort alerts. `call_blocked` independently treats an expired trial as blocked (defense-in-depth) even before the job runs. Sound.
- **requeue_stale_in_progress (M2)** — relies on `CalendarWriteTask.updated_at`, which has `onupdate=func.now()` (schema.py:588-590), so the >5min cutoff measures last-touch, not creation. Correct.
- **CalendarWriteTask.updated_at / FollowupTask.updated_at** — both have server_default + onupdate. No drift vs migrations.
- **Alembic head vs models** — chain is linear (ffcf→8559→fd4a→b7c1→c8paswd→d9clinph→e1diduniq→f2patdemo→g3calllog→h4hardblock), single head; `hard_block_on_exhaust`, all sub-spec A Doctor/Token/FollowupTask columns, token compound indexes, and call_logs index all present in migrations. No model column without a migration found.
- **Admin overview month bucketing (M12/M13)** — month boundary is IST midnight; per-org buckets compare tz-aware `started_at >= this_month` (both aware, correct); the 6-month chart buckets by `started_ist.strftime("%Y-%m")` (IST). Consistent.
- **Analytics call-day buckets (M13)** — `func.date(func.timezone(branch_tz, started_at))` truncates in branch-local tz; minutes-this-month uses the same `_call_day >= month_start`. Consistent with token buckets.
- **Admin/queue/doctors IDOR + super_admin lockout** — every branch-scoped route calls `assert_branch_access` (blocks super_admin + cross-tenant) then filters by branch_id; admin routes are `require_admin` only and expose org-level aggregates, no patient PII. No cross-tenant path found on the new /admin/overview, /admin/orgs/*, /doctors/*.
- **OTP service (M7/M8)** — `verify_code` burns the code and sets a separate `otp_ok:*` flag; register accepts `verify_code OR is_verified` per channel; `otp_echo_enabled` forces off when `app_env=='production'`. Brute force capped at 5, `secrets.compare_digest`. Sound (the only nit: is_verified TTL = 10min, so a very slow form fill re-dead-ends — acceptable).
- **Frontend Admin.jsx controls** — plan `<select>` and hard-block checkbox are query-controlled (value/checked from `ov.clients`), so a failed mutation (no invalidate on error) snaps back to server truth; no optimistic-rollback bug; controlled selects do NOT fire onChange on mount. MoneyTrend/UsageBar/Dashboard charts all guard divisions with `Math.max(...,1)` / ternaries — no divide-by-zero on empty data.
- **Shell.jsx nav** — role→links map gates the menu per role (super_admin sees only Operations, etc.); cosmetic only, backend still enforces.
- **reschedule across doctors** — `_do_reschedule` forces `doctor_id=old_token.doctor_id`; a reschedule can never silently move to a different doctor.
- **_resolve_doctor_id ambiguous name (B6)** — >1 substring match raises ToolError (agent.py:275-283); never silent first-match.

## Notes for the maintainer

T1 is the one that matters: the round-2 C1 fix genuinely closes the sequential
assign-skip double-book, but the headline Rule-2 invariant ("no double-booking,
EVER") still has a concurrency hole on that path because the promised partial
unique index was never created. A DB constraint is the only race-proof fix —
application-level re-counts cannot serialize across independent transactions.

T2 is a "fix shipped but disabled by default" — worth a one-line config change.
